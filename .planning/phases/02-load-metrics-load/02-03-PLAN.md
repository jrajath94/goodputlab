# 02-03: HTTP client + replay engine for trace execution

## Goal

An async HTTP client that executes a `Trace` against an
OpenAI-compatible vLLM endpoint, records per-request telemetry
(LOAD-05 schema: enqueue_ts_ns, ttft_ms, per_token_ts_ns[],
completion_ts_ns, status_code, error, routed_pool), and a replay
wrapper that re-runs a trace from its seed (LOAD-07 contract).

## Why this is its own plan

Generators (02-02) produce *specs* — what to send, when, to whom.
02-03 produces the *telemetry* — what actually happened, with
timestamps.  Plan 02-04 (reconciler) compares the telemetry against
vLLM's `/metrics`; without 02-03 there's nothing to reconcile.

## Client contract

```python
class VllmHttpClient:
    def __init__(
        self,
        base_url: str,
        model: str,
        timeout_s: float = 120.0,
        max_concurrent: int = 64,
    ) -> None: ...

    async def send_one(
        self,
        spec: RequestSpec,
        arrival_ts_ns: int,
        routed_pool: str | None = None,
    ) -> RequestTelemetry: ...
        # Wait until arrival_ts_ns (uses perf_counter_ns as wall clock).
        # POST /chat/completions with stream=true; record TTFT, per-token
        # timestamps, completion, status.  Returns RequestTelemetry.

    async def run(
        self,
        trace: Trace,
        routed_pool_for: Callable[[RequestSpec], str | None] | None = None,
    ) -> list[RequestTelemetry]: ...
        # Walk OpenLoopScheduler(trace); for each (spec, ts), send_one.
        # Concurrency bounded by max_concurrent via asyncio.Semaphore.
```

The client uses `httpx.AsyncClient` with `timeout=timeout_s`.  Streaming
is enabled (`stream=true`) so the first token arrives as soon as the
model emits it (TTFT = wall-clock diff between enqueue and first chunk
parsed), and subsequent tokens are timestamped as they stream.

The "routed_pool" arg is set by the router (Phase 3+).  In Phase 2 we
just pass-through whatever the caller provides (often `"colocated"` or
`None` for direct vLLM hits).

## Replay contract

```python
class ReplayRunner:
    def __init__(self, client: VllmHttpClient) -> None: ...
    async def replay(self, trace: Trace) -> list[RequestTelemetry]: ...
        # Two runs of replay(trace) with the same seed will exercise
        # the same (RequestSpec, arrival_ts_ns) pairs.  Actual telemetry
        # values may differ wall-clock-wise (TTFT, completion_ts_ns
        # depend on server response time), but the *set* of request_ids
        # and the *arrival order* are byte-identical — LOAD-07 contract.
```

## Tests

### `tests/test_client.py`

1. `test_send_one_records_enqueue_ts` — telemetry.enqueue_ts_ns is set
2. `test_send_one_records_ttft` — telemetry.ttft_ms > 0 on a successful stream
3. `test_send_one_records_per_token_ts` — one timestamp per emitted token
4. `test_send_one_records_status_200` — successful response
5. `test_send_one_records_status_error_on_500` — error path
6. `test_send_one_records_error_field_on_exception` — transport failure
7. `test_run_respects_max_concurrent` — semaphore bounds in-flight count
8. `test_run_preserves_request_order_in_output` — list matches scheduler order

### `tests/test_replay.py`

1. `test_replay_produces_telemetry_for_each_request`
2. `test_replay_byte_identical_request_ids` — same trace → same set of IDs in same order (LOAD-07)
3. `test_replay_with_failure_marks_status_code`
4. `test_replay_sets_routed_pool_when_provided` — stub router callback

## Test strategy

- `httpx.MockTransport` is built into httpx.  Tests inject a
  transport handler that returns pre-canned streaming responses
  (chunked-encoded SSE).  No real vLLM, no network.  Runs in CI.
- The mock transport simulates token-by-token emission with a small
  sleep between chunks so TTFT and per-token intervals are measurable.
- For "real" benchmark runs, the same `VllmHttpClient` is used
  unchanged — `base_url` points at vLLM.
